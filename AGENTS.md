# AGENTS.md — ui-studio

This file guides agentic coding agents operating in the `ui-studio` monorepo.

## What this repo is

`ui-studio` is a **pure TypeScript/Bun monorepo** — a meta-tool that generates specialized Claude Code studios for any tech stack. It does **not** contain the Go application it documents; Go patterns live as markdown inside `go-stack-mcp/resources/`.

```
plugins/go-studio/    Claude Code plugin (hooks, rules, agents, skills, workflows)
go-stack-mcp/         MCP server — Go+Templ+HTMX pattern docs
templui-mcp/          MCP server — templUI v1.6 component docs
generator/            MCP server + CLI that generates new studios
```

**Important workspace caveat**: `generator/` is **not** in the root `package.json` workspaces array. It has its own isolated `bun.lock`. Always run `bun install` inside `generator/` separately before working on it.

## Build & Run Commands

### Prerequisites
- **Bun ≥ 1.0.0** (primary runtime — all packages use `bun run`)
- **Node ≥ 22** (for root workspace scripts only)

### Root workspace
```bash
bun install                    # install deps for go-stack-mcp, templui-mcp, plugins/*
npm run build --workspaces     # build all packages (currently no-op — no build scripts)
npm run dev --workspaces       # dev mode for all packages
```

### Individual packages
```bash
# MCP servers — run from within go-stack-mcp/, templui-mcp/, or generator/
bun run index.ts               # start MCP server directly (no compile step)
bun run dev                    # bun --watch index.ts (hot-reload)

# Generator CLI
bun run cli                    # interactive terminal wizard (generator/ only)
```

### Type-checking (no build output)
TypeScript runs **directly via Bun** — there is no `tsc` output, no `dist/` directory.
```bash
# Run from within a package directory (go-stack-mcp/, templui-mcp/, generator/)
bun x tsc --noEmit
```

### Tests
**There are currently no tests in this repo.** When adding tests, use Bun's built-in test runner:
```bash
bun test                        # run all tests in current package
bun test path/to/file.test.ts   # run a single test file
bun test --watch                # watch mode
```

No lint or formatter tooling (no ESLint, Prettier, Biome, or `.editorconfig`). Style is enforced by convention only — see below.

## TypeScript Code Style

### Imports
- Use **named imports** — avoid default imports
- Import types separately with `import type { ... }` — never mix with value imports
- In `.ts` files (Bun), use `import.meta.dir` for the current directory
- In `.mjs` files (Node-compat), use `fileURLToPath(import.meta.url)` instead
- Group order: stdlib → third-party → local (blank lines between groups are optional but consistent within a file)
- Entrypoint files start with a shebang: `#!/usr/bin/env bun`

```typescript
import { readFileSync, existsSync } from 'fs';
import { join } from 'path';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { StackInfo } from './detector';
```

### Dual `.ts` / `.mjs` files
Each MCP entrypoint (`index.ts` / `index.mjs`) and `lib/resources.ts` / `lib/resources.mjs` has a Node-compat twin. The `.mjs` versions replace:
- TypeScript types → JSDoc comments
- `import.meta.dir` → `fileURLToPath(import.meta.url)`
These `.mjs` files are **manually maintained** — they are not generated by a build step. Keep both in sync when making logic changes.

### Interfaces & Types
- Prefer `interface` for object shapes; use `type` for unions and aliases
- All fields must be explicitly typed — no implicit `any`
- Use `[key: string]: string | undefined` index signatures (not `any`)
- Export interfaces used across files; leave file-local ones unexported

```typescript
interface Frontmatter {
  description?: string;
  title?: string;
  [key: string]: string | undefined;
}
```

### Naming Conventions
- **Files**: `kebab-case.ts` — e.g. `session-start.ts`, `resources.ts`, `detector.ts`
- **Interfaces/Types**: `PascalCase` — e.g. `StackInfo`, `InferredPattern`, `StudioConfig`
- **Functions**: `camelCase` — e.g. `detectStack`, `scanDirectory`, `parseFrontmatter`
- **Constants**: `UPPER_SNAKE_CASE` — e.g. `MAX_CHARS_PER_FILE`, `IGNORED_DIRS`, `RESOURCES_DIR`
- **Variables/parameters**: `camelCase`
- Booleans: prefer `is`/`has`/`already` prefix — e.g. `alreadyRegistered`, `isValid`

### Functions
- Top-level functions: named function declarations (`function foo(...)`)
- Callbacks and inline helpers: arrow functions
- All async code uses `async/await` — never `.then()` chains
- Single-responsibility; extract helpers rather than nesting

### Error Handling
- Wrap `server.connect()` and other startup calls in `try/catch`; call `process.exit(1)` on failure
- MCP tools return an error object early instead of throwing:
  ```typescript
  if (!existsSync(project_path)) {
    return { content: [{ type: 'text', text: `❌ project_path does not exist: ${project_path}` }] };
  }
  ```
- CLI top-level: `main().catch(err => { console.error('Fatal:', err); process.exit(1); })`
- Error message extraction: `err instanceof Error ? err.message : String(err)`
- Never swallow errors silently — log to `console.error` at minimum

### Zod Schemas
- All MCP tool inputs validated with inline `zod` schemas on `server.tool()` calls
- Use `.describe()` on every field — it becomes the tool's parameter documentation

```typescript
server.tool('my-tool', { project_path: z.string().describe('Absolute path to the project') }, async ({ project_path }) => { ... });
```

### Formatting
- **2-space indentation** (no tabs)
- **Single quotes** for strings (template literals and JSON are exceptions)
- **Semicolons** present
- Line length: aim for ≤ 100 chars; long template strings in `templates.ts` are acceptable
- Trailing commas in multi-line arrays/objects
- Use `type: 'text' as const` in MCP content arrays
- Section dividers in long files: `// ─── Section name ───────────────────────────`

## Go Code Conventions (for generated client apps)

These apply to the apps this tooling generates, and to Go code in pattern files under `go-stack-mcp/resources/`.

### Package Structure
- `internal/` for all private packages
- `cmd/server/main.go` is the only binary entry point
- One file per feature domain in `handlers/`, `services/`, `repositories/`
- Shared types in `internal/models/`

### Patterns
- **Handlers**: struct-based, take `(w http.ResponseWriter, r *http.Request)`, call service, render templ or redirect
- **Services**: all business logic — no direct DB or HTTP concerns
- **Repositories**: parameterized queries only (no string concat); always `fmt.Errorf("context: %w", err)`; always `defer rows.Close()`

### Error Handling
- Wrap with context: `fmt.Errorf("create user: %w", err)`
- Never swallow errors
- HTTP errors via `http.Error()` or render an error template

### Context
- Always propagate `context.Context` as the first parameter
- Pass `r.Context()` to all DB calls
- Check `ctx.Err()` in long-running loops

### Migrations (Goose)
- SQL files with `-- +goose Up` / `-- +goose Down` markers
- Embed via `//go:embed migrations/*.sql`
- Never modify existing migrations — add new ones

## Plugin & Skill Files

Rules in `plugins/go-studio/rules/*.md` and skills in `plugins/go-studio/skills/*/SKILL.md` are markdown fragments injected into the Claude Code system prompt by `hooks/session-start.ts`. When editing:
- Keep rule fragments focused and scannable — they become system prompt text
- Skill files must have numbered steps and explicit success criteria
- Skill YAML frontmatter must include `description:`, `user-invocable: true`, and `agent:` fields
- Agent files (`agents/*.md`) need only a `description:` frontmatter field
- Resource docs in `go-stack-mcp/resources/` and `templui-mcp/resources/` **must** include YAML frontmatter with a `description` field — it is read by `parseFrontmatter` at server startup
